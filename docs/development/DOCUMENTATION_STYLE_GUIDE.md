# Documentation Style Guide

Standards for writing maintainable, professional documentation in this repository.

## Core rules

1. Prefer task-oriented headings (what to do, how to verify).
2. Keep examples copyable and minimal.
3. State assumptions/prerequisites explicitly.
4. Include verification steps after procedures.
5. Cross-link related docs to reduce dead ends.

## Structure template

- Purpose
- Prerequisites
- Steps
- Validation
- Troubleshooting
- References

## Writing conventions

- Use active voice.
- Keep paragraphs short.
- Use consistent terminology for services and paths.
- Avoid stale placeholders (e.g., generic repo URLs).

## Code/command examples

- Keep commands one per line.
- Indicate environment when needed (local, CI, production).
- Avoid destructive commands without warning labels.

## Versioning and dates

- Update “Last updated” when major edits are made.
- Reflect actual version defaults (e.g., API v2).

## Review checklist

- [ ] Link targets exist
- [ ] Paths and service names are current
- [ ] Steps were validated recently
- [ ] Security-sensitive guidance is accurate
